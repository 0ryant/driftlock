# driftlock — Write-Set Escape Detection (driftlock-writeset-escape demo)

> ADR intent → authorised write-set → agent diff → **escape made visible** → fail-closed gate

Everything here was produced by `driftlock 0.1.0` running on its own fixtures.
No outputs are hand-edited — `findings/` and `results/` are regenerated by
`reproduce.ps1`.

**Synthetic-data disclaimer.** Every artefact in this pack is synthetic. The
`acme-demo` service, ADR-0007, the `session-store` / `billing` crates, the diffs,
and the actor `agent-demo` are fabricated. There are **no real credentials** —
nothing here references any real org, repository, or secret.

## TL;DR

| | `compliant.diff` | `escaping.diff` |
|---|---|---|
| Files touched | 3 | 4 |
| Inside authorised write-set | 3 | 2 |
| **Escaped the write-set** | **0** | **2** |
| `check-diff` verdict (`allowed`) | `true` | `false` |
| `complete` gate exit code | 0 (would pass) | **1 (fail-closed)** |

The two escapes flagged in `escaping.diff` are exactly
`crates/billing/src/charge.rs` and `.github/workflows/release.yml` — both named
non-goals of the work order. The in-scope session-store/test files are not
flagged.

## What happened

1. **What happened.** Synthetic **ADR-0007** decides to add a Redis-backed
   session store. Its derived work order **`adr-0007:T01`** authorises writes to
   exactly two globs:
   - `crates/session-store/**`
   - `tests/session/**`

   ADR-0007 explicitly names billing (`crates/billing/**`) and CI/release config
   as **non-goals**. The authorised boundary lives in
   `input/taskgraph.json` (the `write_set` field of the work order).

2. **What driftlock found (made visible).** Two candidate agent diffs are
   verified against that write-set with `driftlock check-diff`:
   - `compliant.diff` touches 3 files, all inside the write-set →
     `allowed: true`, zero violations.
   - `escaping.diff` touches 4 files. Driftlock flags the **2** that escape the
     authorised boundary:
     - `crates/billing/src/charge.rs` (the billing non-goal)
     - `.github/workflows/release.yml` (the CI/release non-goal)

     and leaves the 2 in-scope files unflagged → `allowed: false`. See
     `findings/escaping.report.json`.

3. **What changed.** Nothing in a real repo — this pack demonstrates detection,
   not remediation. The "change" is that an unbounded ADR-prose-driven diff
   becomes a **bounded, machine-checkable verdict**: scope creep that a human
   reviewer might miss in a multi-file PR is named, path by path.

4. **The gate.** `check-diff` is a reporter: it prints the `DiffReport` and
   always exits 0; the verdict is the `allowed` boolean. The **enforcing** gate
   is `driftlock complete`, which calls the same `verify_changed_files` and
   **bails with exit 1** when `allowed` is false. So a task can never be marked
   complete on a diff that escaped its write-set. The captured transcript is in
   `results/complete-escaping.txt`; the exit-code matrix in
   `results/gate-exit-codes.txt`.

5. **Honest residual.** This pack proves *path-level* write-set enforcement only.
   It does **not** prove:
   - **Sub-file scoping.** Driftlock checks which files changed, not which lines
     or symbols. An in-scope file can still be edited beyond intent and pass.
   - **Diff-source trust.** The verdict is only as honest as the supplied diff /
     `git diff` output. A diff that omits a file hides that file from the check;
     driftlock does not independently re-derive the working-tree change here.
   - **Semantic intent.** A file inside the write-set that implements the *wrong*
     thing is `allowed: true`. Driftlock bounds blast radius; it does not judge
     correctness.
   - **Glob fidelity.** Authorisation is glob-based; an over-broad write-set
     (e.g. `crates/**`) would legitimately admit billing. The strength of the
     gate is the strength of the ADR-derived write-set, which is upstream of this
     check.

## Reproduce it

```pwsh
pwsh -File docs/demo/driftlock-writeset-escape/reproduce.ps1
```

That script (a) drift-checks the vendored `_assets/site.css` against the
algol.cc source sha256, (b) builds `driftlock-cli` into an isolated
`target-ci/`, (c) re-runs `check-diff` on both diffs into `findings/`, and
(d) runs `complete` on the escaping diff to capture the fail-closed gate. Or by
hand:

```pwsh
$bin = "target-ci\release\driftlock.exe"
$g   = "docs\demo\driftlock-writeset-escape\input\taskgraph.json"
& $bin check-diff --graph $g --task adr-0007:T01 --diff-file ...\compliant.diff <pack>
& $bin check-diff --graph $g --task adr-0007:T01 --diff-file ...\escaping.diff  <pack>
& $bin complete   --graph $g --task adr-0007:T01 --actor agent-demo --diff-file ...\escaping.diff <pack>
```

*Generated with driftlock 0.1.0. The ADR, work order, diffs, crates, service, and
actor are synthetic; the credentials are fake placeholders.*
